# 限流与防重提交使用指南

## 前言：两个注解分别解决什么问题

假设你正在做一个后台管理系统，有两个常见的问题：

**问题一：有人恶意刷接口**

一个用户用脚本在 1 秒内调用了 1000 次「发送短信验证码」接口。这会导致短信费用爆炸，服务器压力骤增。

**问题二：用户手抖点了两次提交**

用户填写了一个表单，点击「保存」按钮。页面有点卡，他又点了一次。结果数据库里出现了两条一模一样的数据。

这两个问题对应了两个注解：

| 注解 | 解决问题 | 类比 |
|------|----------|------|
| `@RateLimiter` | 限制调用频率（同一接口，N秒内最多调用M次） | 地铁闸机——每分钟只能通过一定人数 |
| `@RepeatSubmit` | 防止相同请求被重复处理（同一请求，N秒内只能提交一次） | 排队叫号——同一个号只能办一次业务 |

## 第一章：@RateLimiter 限流注解

### 1.1 基本原理

`@RateLimiter` 使用 **Redisson 的令牌桶算法**（RRateLimiter）。

想象一个水桶，系统以固定速率往桶里放令牌（比如每秒放 1 个）。每次请求需要从桶里拿走一个令牌。如果桶空了，请求就被拒绝。

```
时间 →
  放令牌:  ●    ●    ●    ●    ●    ●
  取令牌:   ●         ●●(桶空了！拒绝)
```

**和固定窗口的区别**：固定窗口在窗口边界处会有"突刺"问题。比如限制每分钟 60 次，用户可以在第 59 秒发 60 次，第 61 秒又发 60 次——实际 2 秒内发了 120 次。令牌桶不会这样。

### 1.2 基本用法

```java
@RestController
public class DemoController {

    /**
     * 全局限流：60秒内所有用户合计最多调用100次
     */
    @RateLimiter(time = 60, count = 100)
    @GetMapping("/demo/global")
    public Result<String> globalLimit() {
        return Result.ok("ok");
    }
}
```

触发限流时，会直接抛异常并返回：
```json
{
  "code": 201,
  "message": "访问过于频繁，请稍候再试"
}
```

### 1.3 按 IP 限流

短信验证码场景：同一个 IP 在 60 秒内只能发送 1 次验证码。

```java
/**
 * limitType = LimitType.IP：每个IP独立计数
 * 张三的IP发了1次被限流，不影响李四的IP正常发送
 */
@RateLimiter(time = 60, count = 1, limitType = LimitType.IP)
@GetMapping("/sms/code")
public Result<Void> sendSmsCode(String phone) {
    // 发送短信验证码
    return Result.ok();
}
```

**为什么用 IP 而不是手机号？** 因为手机号是用户输入的，恶意用户可以换不同的手机号继续刷。IP 限流增加了攻击成本。

### 1.4 使用 SpEL 动态 Key

有时候需要根据请求参数来限流。比如「同一个手机号 60 秒内只能发 1 次」：

```java
/**
 * #phone 是方法参数名，SpEL 表达式会自动取它的值
 * Redis key 变成: rate_limit:/sms/code:13800138000
 * 不同的手机号生成不同的 key，互不影响
 */
@RateLimiter(key = "#phone", time = 60, count = 1)
@GetMapping("/sms/code")
public Result<Void> sendSmsCode(@RequestParam String phone) {
    return Result.ok();
}
```

SpEL 也支持嵌套取属性：

```java
// 取对象的属性
@RateLimiter(key = "#user.userId", time = 60, count = 5)
@PostMapping("/save")
public Result<Void> save(@RequestBody UserDTO user) { ... }

// 取多个参数组合
@RateLimiter(key = "#userId + ':' + #action", time = 60, count = 3)
@GetMapping("/operate")
public Result<Void> operate(String userId, String action) { ... }
```

### 1.5 自定义提示消息

```java
@RateLimiter(
    key = "#phone",
    time = 60,
    count = 1,
    message = "短信发送太频繁了，请1分钟后再试"
)
@GetMapping("/sms/code")
public Result<Void> sendSmsCode(String phone) { ... }
```

### 1.6 完整参数说明

```java
@RateLimiter(
    key = "#phone",          // 限流key，支持SpEL表达式。默认""表示按URL全局限流
    time = 60,               // 时间窗口（秒），默认60
    count = 100,             // 窗口内最大请求次数，默认100
    limitType = LimitType.IP,// 限流类型：DEFAULT(全局) / IP(按请求IP)
    message = "访问过于频繁"   // 触发限流时的提示消息
)
```

## 第二章：@RepeatSubmit 防重提交注解

### 2.1 基本原理

`@RepeatSubmit` 使用 **Redis SETNX**（SET if Not eXists）实现分布式锁。

```
请求到达
  │
  ▼
计算签名 = MD5(Token + URL + 请求参数)
  │
  ▼
Redis SETNX(key=签名, TTL=5秒)
  │
  ├── 成功（key不存在，首次请求）
  │     → 放行，执行方法
  │     → 方法成功：保留key（5秒内不允许重复提交）
  │     → 方法失败或抛异常：删除key（允许重试）
  │
  └── 失败（key已存在，重复提交）
        → 拒绝，抛 ServiceException
```

**关键设计**：
- 签名中包含了 Token，所以不同用户的相同请求不会互相误判
- 方法返回成功才保留 key，失败则删除 key 允许重试
- key 有 TTL 过期时间，不会永久占用 Redis 内存

### 2.2 基本用法

```java
/**
 * 5秒内不允许重复提交相同的表单数据
 */
@RepeatSubmit(interval = 5000)
@PostMapping("/notice/save")
public Result<Void> save(@RequestBody SysNotice notice) {
    noticeService.save(notice);
    return Result.ok();
}
```

触发防重时的返回：
```json
{
  "code": 201,
  "message": "不允许重复提交，请稍候再试"
}
```

### 2.3 调整间隔时间

```java
// 3秒内不能重复提交
@RepeatSubmit(interval = 3000)
@PostMapping("/save")
public Result<Void> save(@RequestBody DataDTO data) { ... }

// 使用 TimeUnit 指定单位（更直观）
@RepeatSubmit(interval = 1, timeUnit = TimeUnit.MINUTES)
@PostMapping("/important/save")
public Result<Void> importantSave(@RequestBody DataDTO data) { ... }
```

### 2.4 自定义提示消息

```java
@RepeatSubmit(
    interval = 5000,
    message = "请勿重复提交，5秒后再试"
)
@PostMapping("/save")
public Result<Void> save(@RequestBody DataDTO data) { ... }
```

### 2.5 什么情况下 key 会被自动删除

| 场景 | key 处理 | 原因 |
|------|----------|------|
| 方法正常返回 `Result.ok()` | **保留** key | 防止间隔内重复提交，key 到期后自动删除 |
| 方法返回 `Result.fail()` | **删除** key | 业务失败，允许用户修改后重试 |
| 方法抛出任何异常 | **删除** key | 系统异常，允许用户重试 |
| key 达到 TTL 时间 | **自动过期** | Redis 的 TTL 机制，不占用内存 |

### 2.6 完整参数说明

```java
@RepeatSubmit(
    interval = 5000,                     // 间隔时间
    timeUnit = TimeUnit.MILLISECONDS,     // 时间单位（默认毫秒）
    message = "不允许重复提交，请稍候再试"  // 提示消息
)
```

## 第三章：两个注解组合使用

同一个方法可以同时加两个注解，实现"限流 + 防重"双层保护：

```java
/**
 * 导入文件场景：
 * - @RateLimiter: 同一用户每分钟最多导入3次
 * - @RepeatSubmit: 相同文件内容5秒内不允许重复提交
 */
@RateLimiter(key = "#userId", time = 60, count = 3)
@RepeatSubmit(interval = 5000)
@PostMapping("/import")
public Result<Void> importData(String userId, MultipartFile file) {
    // 导入逻辑
    return Result.ok();
}
```

**执行顺序**：
1. 先经过 `@RateLimiter` 检查 → 超限则拒绝，不超限继续
2. 再经过 `@RepeatSubmit` 检查 → 重复则拒绝，不重复继续
3. 执行方法体

## 第四章：实战场景

### 4.1 短信验证码

```java
@RateLimiter(key = "#phone", time = 60, count = 1)
@GetMapping("/resource/sms/code")
public Result<Void> sendSmsCode(@RequestParam String phone) {
    String code = RandomUtil.randomNumbers(4);
    redisCacheUtils.setCacheObject("sms:" + phone, code, 5, TimeUnit.MINUTES);
    // 调用短信服务发送
    return Result.ok();
}
```

效果：每个手机号 60 秒内只能发 1 次。换不同手机号不受限。

### 4.2 表单提交

```java
@RepeatSubmit(interval = 3000)
@PostMapping("/notice/save")
public Result<Void> saveNotice(@RequestBody SysNotice notice) {
    notice.setCreateTime(new Date());
    noticeService.save(notice);
    return Result.ok();
}
```

效果：用户 3 秒内点击两次提交按钮，第二次被拦截。

### 4.3 登录接口

```java
@RateLimiter(key = "#loginDTO.phone", time = 60, count = 5)
@PostMapping("/user/login")
public Result<String> login(@RequestBody loginDTO loginDTO) {
    return Result.ok(loginService.login(loginDTO));
}
```

效果：同一手机号 60 秒内最多尝试登录 5 次，防止暴力破解。

### 4.4 文件上传

```java
@RateLimiter(time = 60, count = 10, limitType = LimitType.IP)
@RepeatSubmit(interval = 10000)
@PostMapping("/file/upload")
public Result<String> upload(MultipartFile file) {

    return Result.ok(url);
}
```

效果：限制每个 IP 每分钟上传 10 次 + 相同文件 10 秒内不重复。

## 第五章：常见问题

### Q1: 限流和防重有什么区别？

| | @RateLimiter | @RepeatSubmit |
|---|---|---|
| 目的 | 控制调用频率 | 防止相同请求重复处理 |
| 判断依据 | 固定时间窗口内的总次数 | 相同签名的请求是否已存在 |
| 不同请求 | 累加计数 | 互不影响（签名不同） |
| 计数器重置 | 时间窗口结束后自动重置 | key 过期或方法失败后删除 |

### Q2: 为什么防重用 MD5 而不是直接用参数？

请求参数可能很长（比如一个富文本编辑器的 HTML 内容），直接拼接到 Redis key 会导致 key 过长。MD5 把任意长度的内容压缩成固定 32 字符的哈希值。

### Q3: 防重提交对文件上传有效吗？

有效但有限。代码中已经过滤了 `MultipartFile` 对象——文件内容不参与签名计算（否则每次签名都不同，无法防重）。签名只包含 Token + URL + 其他字符串参数。如果要精确防重文件上传，可以额外传一个文件的 MD5 值作为参数。

### Q4: 分布式环境下防重有效吗？

有效。Redis 本身是共享的，所有实例连接到同一个 Redis，SETNX 在分布式环境下天然安全。

### Q5: 限流的令牌桶和固定窗口哪个好？

令牌桶更好。固定窗口在时间窗口边界会有"突刺"问题。Redisson 的 RRateLimiter 底层基于 Redis 的 Hash + Lua 脚本，性能和准确度都很好。

### Q6: SpEL 表达式怎么写？

| 需求 | 写法 |
|------|------|
| 取方法参数值 | `#参数名`，如 `#phone` |
| 取对象属性 | `#对象.属性`，如 `#user.userId` |
| 字符串拼接 | `#a + ':' + #b` |
| 取请求体中的字段 | `#dto.field` |

**注意**：SpEL 中的参数名必须和方法的参数名完全一致。Java 编译时默认不保留参数名。确保在 pom.xml 中开启了 `-parameters` 编译选项（Spring Boot 默认开启）。

### Q7: 怎么知道限流是否生效？

看日志：

```
# 正常放行
限流通行: key=rate_limit:/sms/code:13800138000, 限制1/60秒

# 触发限流
限流触发: key=rate_limit:/sms/code:13800138000, 限制1/60秒

# 重复提交
重复提交: url=/notice/save
```
