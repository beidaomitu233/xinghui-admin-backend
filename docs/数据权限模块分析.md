# 数据权限模块实现详解

## 一、总览：什么是数据权限

### 1.1 问题背景

在一个多用户后台管理系统中，不同角色的用户应当看到不同范围的数据：

| 角色 | 数据范围 | 示例 |
|------|----------|------|
| 超级管理员 | 全部数据 | `SELECT * FROM sys_notice` |
| 租户管理员 | 本租户全部数据 | `SELECT * FROM sys_notice WHERE tenant_id = '000000'` |
| 普通用户 | 仅自己创建的数据 | `SELECT * FROM sys_notice WHERE create_by = 'user123'` |

**租户隔离**由多租户插件（TenantLineInnerInterceptor）处理，在 SQL 层自动追加 `tenant_id = 'xxx'`。

**数据权限**解决的是同一个租户内「谁能看谁的」问题——通过判断当前用户的角色，决定是否在 SQL 中追加 `create_by = 'userId'` 条件。

### 1.2 核心设计思想

本实现借鉴 RuoYi-Vue-Plus 的数据权限架构，但做了大幅简化：

| 对比维度 | RuoYi-Vue-Plus | 我们的实现 |
|----------|---------------|-----------|
| 数据范围类型 | 5种（ALL/CUSTOM/DEPT/DEPT_AND_CHILD/SELF） | 2种（ALL/SELF） |
| SQL 模板引擎 | SpEL 表达式 | 直接拼接 JSqlParser AST |
| 部门支持 | 完整的部门树查询服务 | 无部门模块 |
| 登录上下文 | Sa-Token LoginHelper | Spring Security SecurityUtils |
| 权限标识符 | 支持 perms 跳过 | 暂不支持 |

**核心原则**：AOP 捕获注解 → ThreadLocal 传递上下文 → MyBatis 拦截器注入 SQL 条件。

### 2.3 拦截器顺序

在 `mybatisConfig` 中，拦截器按以下顺序注册：

```java
// 1. 多租户插件（第一位，在所有其他拦截器之前执行）
tenantProvider.ifAvailable(interceptor::addInnerInterceptor);

// 2. 数据权限插件（第二位）
interceptor.addInnerInterceptor(new PlusDataPermissionInterceptor());

// 3. 分页插件（第三位）
interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
```

**为什么租户插件必须第一位？** MyBatis-Plus 的拦截器链按顺序处理 SQL。租户插件需要先追加 `tenant_id` 条件，确保后续的数据权限过滤是在正确的租户数据子集上操作。如果顺序颠倒，数据权限先过滤，租户条件后追加可能导致跨租户数据泄露。

## 三、各层组件详解

### 3.1 注解层（common-core）

注解层定义了数据权限的**声明式 API**——开发者通过在 Mapper 上添加注解来声明数据权限规则，无需写任何 SQL。

#### 3.1.1 @DataPermission

```java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface DataPermission {
    DataColumn[] value();  // 列映射配置数组
}
```

**设计意图**：
- 可标注在 Mapper 接口的方法上（精确控制）
- 也可标注在 Mapper 接口类上（批量生效）
- `value()` 返回数组，支持多列过滤

**为什么是数组？** 某些场景需要同时过滤多列。例如一个 Mapper 同时有 `create_by` 和 `update_by`，可以配置两个 DataColumn 实现 OR 逻辑。

#### 3.1.2 @DataColumn

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DataColumn {
    String key() default "userId";       // 占位符名称
    String value() default "create_by";  // 数据库列名
}
```

**设计意图**：
- `key` 是抽象占位符——在 SQL 模板中引用
- `value` 是实际的数据库列名——最终拼入 SQL

**为什么需要映射层？** 解耦注解声明和数据库实现。如果数据库列名变了（如 `create_by` → `creator_id`），只需改注解的 `value`，无需改任何 Java 逻辑。

#### 3.1.3 DataScopeType

```java
public enum DataScopeType {
    ALL("1", "全部数据权限"),    // 不过滤
    SELF("5", "仅本人数据权限");  // WHERE create_by = userId
}
```

**为什么只有两种？** 我们没有部门模块，CUSTOM/DEPT/DEPT_AND_CHILD 三种范围无意义。保留 `ALL` 和 `SELF` 两种最通用的类型，后续如需扩展只需加枚举值和对应的 SQL 模板。

`code` 值与数据库 `sys_role.data_scope` 字段对应：
- `"1"` = ALL
- `"5"` = SELF

### 3.2 上下文管理层（DataPermissionHelper）

`DataPermissionHelper` 是整个数据权限系统的**中枢**——它使用 ThreadLocal 在当前线程内传递 `@DataPermission` 注解。

#### 3.2.1 核心数据结构

```java
// 存储当前 Mapper 方法上的 @DataPermission 注解
private static final ThreadLocal<DataPermission> PERMISSION_CACHE = new ThreadLocal<>();

// 可重入忽略栈：支持嵌套的 ignore() 调用
private static final ThreadLocal<Stack<Integer>> REENTRANT_IGNORE = 
    ThreadLocal.withInitial(Stack::new);
```

**为什么用 ThreadLocal？** MyBatis 的 Mapper 执行和 SQL 拦截发生在同一个线程内。AOP 在方法入口设置注解，MyBatis 拦截器在 SQL 执行时读取，全程在同一个线程，ThreadLocal 是最轻量的传递方式。

**为什么 PERMISSION_CACHE 不存 Map 而只存单个注解？** 参考项目用 Sa-Token 的 SaStorage（类似 Session），可以存 Map 配 SpEL 变量。我们简化后每次只有一个 `@DataPermission` 注解在生效，无需 Map。

#### 3.2.2 可重入忽略机制

```java
private static final ThreadLocal<Stack<Integer>> REENTRANT_IGNORE = ...;

public static void ignore(Runnable handle) {
    enableIgnore();           // push 到栈
    try { handle.run(); }
    finally { disableIgnore(); }  // pop 出栈
}
```

**为什么用栈？** 支持嵌套调用：
```java
DataPermissionHelper.ignore(() -> {
    tenantService.list();           // 不启用数据权限
    DataPermissionHelper.ignore(() -> {
        userService.list();         // 仍然不启用（栈深度 2）
    });
    // 回到栈深度 1，仍然不启用
});
// 栈为空，恢复数据权限
```

只有最外层 `ignore()` 退出时（栈变空）才真正恢复数据权限，防止内层提前恢复导致外层数据泄露。

#### 3.2.3 ignore() 的实现原理

```java
private static void enableIgnore() {
    // 获取 MyBatis-Plus 内部的全局 IgnoreStrategy
    IgnoreStrategy strategy = getIgnoreStrategy();
    if (strategy == null) {
        // 首次创建 Strategy，只设置 dataPermission=true
        InterceptorIgnoreHelper.handle(
            IgnoreStrategy.builder().dataPermission(true).build());
    } else {
        // 已有 Strategy（可能由租户模块创建），追加 dataPermission=true
        strategy.setDataPermission(true);
    }
    REENTRANT_IGNORE.get().push(size + 1);
}
```

**关键点**：借助 MyBatis-Plus 内置的 `InterceptorIgnoreHelper` 机制。当 `IgnoreStrategy.dataPermission = true` 时，数据权限拦截器会自动跳过当前 SQL。这与租户模块的 `TenantHelper.ignore()` 使用的是同一套底层 API，共享同一个 `IgnoreStrategy` 对象（通过反射从 MyBatis-Plus 的私有 ThreadLocal 中获取）。

### 3.3 AOP 层

AOP 层负责**在 Mapper 方法被调用前**捕获其上的 `@DataPermission` 注解，并存入 `DataPermissionHelper`。

#### 3.3.1 DataPermissionPointcut（切入点）

```java
public class DataPermissionPointcut extends StaticMethodMatcherPointcut {
    @Override
    public boolean matches(Method method, Class<?> targetClass) {
        // 1. 方法上有 @DataPermission
        if (method.isAnnotationPresent(DataPermission.class)) return true;
        // 2. 类上有 @DataPermission
        if (targetClass.isAnnotationPresent(DataPermission.class)) return true;
        // 3. Mapper 接口方法上有 @DataPermission
        for (Class<?> iface : targetClass.getInterfaces()) {
            // 匹配接口上的注解
        }
        return false;
    }
}
```

**为什么要检查三层？**

```
// 场景1: 方法级注解（最精确）
public interface NoticeMapper {
    @DataPermission({@DataColumn(key="userId", value="create_by")})
    List<SysNotice> selectList();
}

// 场景2: 类级注解（批量生效）
@DataPermission({@DataColumn(key="userId", value="create_by")})
public interface NoticeMapper {
    List<SysNotice> selectList();    // 继承类上的注解
    int update(SysNotice notice);     // 继承类上的注解
}

// 场景3: MyBatis 代理类
// MyBatis 为 Mapper 接口生成 JDK 动态代理
// targetClass = com.sun.proxy.$Proxy123
// 实际接口 = NoticeMapper.class
// 需要从接口方法上获取注解
```

`StaticMethodMatcherPointcut` 是 Spring AOP 提供的基类——匹配结果可缓存，性能优于每次动态计算。

#### 3.3.2 DataPermissionAdvice（通知）

```java
public class DataPermissionAdvice implements MethodInterceptor {
    @Override
    public Object invoke(MethodInvocation invocation) throws Throwable {
        // 1. 获取注解
        DataPermission annotation = getAnnotation(target, method);
        // 2. 存入 ThreadLocal
        DataPermissionHelper.setPermission(annotation);
        try {
            // 3. 执行原始方法（SQL 会在此过程中被 MyBatis 拦截器改写）
            return invocation.proceed();
        } finally {
            // 4. 清理 ThreadLocal（防止内存泄漏和上下文污染）
            DataPermissionHelper.removePermission();
        }
    }
}
```

**为什么用 MethodInterceptor 而不是 @Around 注解？**

`MethodInterceptor` 是 AOP Alliance 的标准接口（Spring AOP 底层依赖），配合 `AbstractPointcutAdvisor` 可以实现**编程式 AOP**而非声明式。好处：

1. **零反射开销**：不依赖 `@Around` 注解的运行时解析
2. **精确控制**：Advice 只在 Pointcut 匹配时才执行，不会有多余代理
3. **基础设施语义**：通过 `@Role(BeanDefinition.ROLE_INFRASTRUCTURE)` 标记为框架内部组件

#### 3.3.3 DataPermissionPointcutAdvisor（组装器）

```java
public class DataPermissionPointcutAdvisor extends AbstractPointcutAdvisor {
    private final Advice advice = new DataPermissionAdvice();
    private final Pointcut pointcut = new DataPermissionPointcut();
}
```

Advisor = Pointcut（在哪切） + Advice（切了做什么）。Spring AOP 会将此 Advisor 注册到代理创建流程中，对所有匹配 Pointcut 的 Mapper Bean 自动创建 AOP 代理。

#### 3.3.4 AOP 执行时序

```

        │
        ▼

        │
        ▼
DataPermissionAdvice.invoke()
  ├── 1. 读取 @DataPermission 注解

  ├── 3. invocation.proceed() ──────────────────────────┐
  │                                                      │
```

### 3.4 SQL 处理层

#### 3.4.1 PlusDataPermissionHandler（核心逻辑）

这是整个系统的**大脑**——根据当前用户角色决定 SQL 过滤策略。

```java
public Expression getSqlSegment(Expression where, boolean isSelect) {
    // 步骤1: 获取 @DataPermission 注解
    DataPermission dataPermission = DataPermissionHelper.getPermission();
    if (dataPermission == null) return where;  // 无注解，不处理

    // 步骤2: 获取当前登录用户
    loginUser loginUser = SecurityUtils.getLoginUser();
    if (loginUser == null) return where;  // 未登录，不处理

    // 步骤3: 超管或租户管理员 → 不过滤
    if (isSuperAdmin(loginUser) || isTenantAdmin(loginUser)) {
        return where;
    }

    // 步骤4: 获取角色数据范围
    DataScopeType scopeType = getDataScopeType(loginUser);
    if (scopeType == DataScopeType.ALL) return where;  // 全部权限，不过滤

    // 步骤5: 构建过滤条件
    // SELF → create_by = 'currentUserId'
    Expression filterExpr = buildFilterExpression(...);

    // 步骤6: 用括号包裹，与原 WHERE 用 AND 连接
    // 原 WHERE: status = '0'
    // 新 WHERE: status = '0' AND (create_by = 'user123')
    return new AndExpression(where, parenthesized(filterExpr));
}
```

**关键实现细节**：

##### 角色判断

```java
private boolean isSuperAdmin(loginUser loginUser) {
    // 检查用户的权限列表中是否包含 superadmin
    return loginUser.getPermissions().contains("superadmin");
}

private boolean isTenantAdmin(loginUser loginUser) {
    return loginUser.getPermissions().contains("admin");
}
```

这里不查数据库而是直接检查登录时已加载的权限列表（`loginUser.permissions`），因为 roleKey 在登录时已经作为权限标识存入。这避免了每次 SQL 执行都查数据库。

##### SQL 条件构建（不使用 SpEL）

```java
private Expression buildFilterExpression(...) throws JSQLParserException {
    // SELF 模式：create_by = 'userId'
    String columnName = col.value();           // "create_by"
    String userId = loginUser.getUser().getUserId();  // "abc123-def456"
    return new EqualsTo(
        new Column(columnName),
        new StringValue(userId)
    );
}
```

**为什么不用 SpEL？** 参考项目用 SpEL 是为了支持复杂的 SQL 模板（如 `#{@sdss.getDeptAndChild(#user.deptId)}`）。我们的场景简单得多——只有 `create_by = 'xxx'` 一种模板，直接用 JSqlParser 的 AST API（`EqualsTo`、`Column`、`StringValue`）构建更简单、更安全：

- SpEL 需要模板解析器、变量上下文、Bean 解析器 → 200+ 行代码
- JSqlParser AST 直接构建 → 3 行代码
- SpEL 字符串拼接有 SQL 注入风险
- JSqlParser 的 `StringValue` 自动处理引号转义

#### 3.4.2 PlusDataPermissionInterceptor（拦截器）

拦截器继承 `BaseMultiTableInnerInterceptor`，这是 MyBatis-Plus 提供的基类，封装了 SELECT/UPDATE/DELETE 的 SQL 解析逻辑。

```java
public class PlusDataPermissionInterceptor extends BaseMultiTableInnerInterceptor {
    private final PlusDataPermissionHandler handler = new PlusDataPermissionHandler();

    // ========== 查询拦截 ==========
    @Override
    public void beforeQuery(...) {
        if (InterceptorIgnoreHelper.willIgnoreDataPermission(ms.getId())) {
            return;  // 被 ignore() 包裹的调用，跳过
        }
        if (handler.invalid()) {
            return;  // 当前 Mapper 没有 @DataPermission 注解
        }
        mpBs.sql(parserSingle(mpBs.sql(), ms.getId()));  // 改写 SQL
    }

    // ========== 更新/删除拦截 ==========
    @Override
    public void beforePrepare(...) {
        if (sct == UPDATE || sct == DELETE) {
            // 同理，对 UPDATE/DELETE 也追加 WHERE 条件
            mpBs.sql(parserMulti(mpBs.sql(), ms.getId()));
        }
    }

    // ========== SELECT 的 WHERE 注入 ==========
    @Override
    protected void processSelect(Select select, ...) {
        Expression segment = handler.getSqlSegment(plainSelect.getWhere(), true);
        plainSelect.setWhere(segment);  // 替换 WHERE 子句
    }

    // ========== UPDATE 的 WHERE 注入 ==========
    @Override
    protected void processUpdate(Update update, ...) {
        Expression segment = handler.getSqlSegment(update.getWhere(), false);
        update.setWhere(segment);
    }

    // ========== DELETE 的 WHERE 注入 ==========
    @Override
    protected void processDelete(Delete delete, ...) {
        Expression segment = handler.getSqlSegment(delete.getWhere(), false);
        delete.setWhere(segment);
    }
}
```

**为什么 UPDATE/DELETE 也要拦截？** 如果不拦截，普通用户可以通过不带 WHERE 的 UPDATE 修改别人的数据：

```sql
-- 危险！不拦截会更新所有人的通知
UPDATE sys_notice SET status = '1'

-- 拦截后变成：
UPDATE sys_notice SET status = '1' WHERE (create_by = 'user123')
```

#### 3.4.3 查询 vs 更新的连接符差异

```java
// 查询：不同角色的条件用 OR 连接（宽松）
// 角色A 的 SELF 和 角色B 的 SELF → OR
SELECT * FROM notice WHERE (create_by = 'userA' OR create_by = 'userB')

// 更新/删除：不同角色的条件用 AND 连接（严格）
// 防止越权操作
UPDATE notice SET status='1' WHERE (create_by = 'userA' AND create_by = 'userB')
```

### 3.5 配置集成（mybatisConfig）

```java
@Configuration
public class mybatisConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor(
            ObjectProvider<TenantLineInnerInterceptor> tenantProvider) {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        // 1. 多租户（第一位）—— 过滤租户
        tenantProvider.ifAvailable(interceptor::addInnerInterceptor);

        // 2. 数据权限（第二位）—— 过滤用户
        interceptor.addInnerInterceptor(new PlusDataPermissionInterceptor());

        // 3. 分页（第三位）—— 在过滤后的数据集上分页
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));

        return interceptor;
    }

    @Bean
    @Role(BeanDefinition.ROLE_INFRASTRUCTURE)
    public DataPermissionPointcutAdvisor dataPermissionPointcutAdvisor() {
        return new DataPermissionPointcutAdvisor();
    }
}
```

**`@Role(BeanDefinition.ROLE_INFRASTRUCTURE)` 的作用**：标记此 Bean 为框架基础设施组件。在 Spring AOP 的代理创建过程中，基础设施 Advisor 的优先级高于用户 Advisor，且不会被 AOP 自身代理（避免无限递归）。

## 四、完整执行流程

以一个普通用户查询通知列表为例：

```
步骤1: Controller 调用
  noticeService.list() → noticeMapper.selectList()

步骤2: AOP 拦截（DataPermissionAdvice）
  检测到 NoticeMapper 上有 @DataPermission({@DataColumn(key="userId", value="create_by")})
  → DataPermissionHelper.setPermission(注解)

步骤3: MyBatis 准备执行 SQL
  SELECT * FROM sys_notice WHERE status = '0'

步骤4: MyBatis 拦截器链

  4a. TenantLineInnerInterceptor（第一位）
       TenantHelper.getTenantId() → "000000"
       SQL → SELECT * FROM sys_notice WHERE tenant_id = '000000' AND status = '0'

  4b. PlusDataPermissionInterceptor（第二位）
       beforeQuery() 被调用
       → InterceptorIgnoreHelper.willIgnoreDataPermission() → false（未忽略）
       → handler.invalid() → false（有 @DataPermission 注解）
       → handler.getSqlSegment(where, isSelect=true):
           ┌─ DataPermissionHelper.getPermission() → @DataPermission 注解
           ├─ SecurityUtils.getLoginUser() → loginUser (角色: common, userId: user123)
           ├─ isSuperAdmin() → false
           ├─ isTenantAdmin() → false
           ├─ getDataScopeType() → DataScopeType.SELF
           └─ buildFilterExpression() → EqualsTo(Column("create_by"), StringValue("user123"))
       → 原始 WHERE: tenant_id = '000000' AND status = '0'
       → 新 WHERE:   tenant_id = '000000' AND status = '0' AND (create_by = 'user123')

  4c. PaginationInnerInterceptor（第三位）
       在已过滤的数据上追加 LIMIT/OFFSET

步骤5: 最终 SQL 送到数据库
  SELECT * FROM sys_notice
  WHERE tenant_id = '000000'
    AND status = '0'
    AND (create_by = 'user123')
  LIMIT 0, 10

步骤6: 结果返回后
  DataPermissionHelper.removePermission()（finally 块保证清理）

结果：普通用户只看到自己创建的通知，看不到同一租户内其他用户的。
```

## 五、方法级 vs 类级注解

```java
// ====== 类级注解：所有方法都继承 ======
@DataPermission({@DataColumn(key = "userId", value = "create_by")})
public interface SysNoticeMapper extends BaseMapper<SysNotice> {
    // selectList()  → 注入 create_by 过滤
    // selectById()  → 注入 create_by 过滤
    // updateById()  → 注入 create_by 过滤
    // deleteById()  → 注入 create_by 过滤
}

// ====== 方法级注解：只影响特定方法 ======
public interface SysNoticeMapper extends BaseMapper<SysNotice> {
    @DataPermission({@DataColumn(key = "userId", value = "create_by")})
    List<SysNotice> selectNoticeList();  // 只有这个方法被过滤
    SysNotice selectById(Long id);       // 这个方法不过滤
}
```

## 六、如何给角色配置数据范围

1. 在 `sys_role` 表中设置 `data_scope` 字段：
   - `"1"` = 全部数据
   - `"5"` = 仅本人

2. 如果要给用户分配「全部数据」权限，将其角色设为超级管理员：
   ```
   sys_role: role_key = "superadmin", data_scope = "1"
   ```
   或在 `sys_role` 中任意角色设置 `data_scope = "1"`

3. 普通角色默认 `data_scope = "5"`（仅本人），通过注解的 @DataPermission 控制。

## 七、关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 模板引擎 | JSqlParser AST（非 SpEL） | 场景简单，AST 类型安全，无注入风险 |
| 上下文传递 | ThreadLocal | 单线程执行，零网络开销 |
| AOP 方式 | MethodInterceptor（非 @Around） | 编程式精确控制，基础设施语义 |
| 角色判断 | 检查 permissions 列表（非查库） | 登录时已加载，避免每次 SQL 都查库 |
| 拦截器基类 | BaseMultiTableInnerInterceptor | 自动处理 SELECT/UPDATE/DELETE 的 SQL 解析 |
| 数据范围 | 2 种（ALL/SELF），无部门 | 无部门模块，保持简洁 |
