# 数据权限模块 — 从零实现教程

## 前言：这篇教程适合谁

这篇教程假设你：
- 会用 Spring Boot + MyBatis-Plus 写 CRUD
- 了解 `@PreAuthorize` 的基本用法
- 不知道"数据权限"是什么、怎么实现

读完这篇教程，你将：
- 理解功能权限和数据权限的区别
- 能从头实现一个基于角色的数据权限系统
- 知道如何在添加新角色时配置数据范围

---

## 第一章：先搞清楚为什么要做这个

### 1.1 一个真实的场景

假设你正在做一个后台管理系统，有一个「通知公告」页面。系统里有三个用户：

| 用户 | 角色 | 能看到哪些通知？ |
|------|------|-----------------|
| 张三 | 超级管理员 | **全部** 通知 |
| 李四 | 普通员工 | 只能看**自己发的**通知 |
| 王五 | 普通员工 | 只能看**自己发的**通知 |

数据库里有 5 条通知：

```
notice_id | title          | create_by
----------|----------------|----------
1         | 放假通知        | 张三
2         | 会议通知        | 李四
3         | 系统维护通知    | 张三
4         | 报销流程变更    | 王五
5         | 团建通知        | 李四
```

**期望结果：**

- 张三登录，看到 5 条（全部）
- 李四登录，看到 2 条（notice_id=2 和 5）
- 王五登录，看到 1 条（notice_id=4）

### 1.2 不用数据权限，你会怎么做？

最直觉的写法——在每个接口里手动判断：

```java
@GetMapping("/list")
public Result<List<SysNotice>> list() {
    List<SysNotice> notices;
    if (当前用户是超管) {
        notices = noticeService.list();  // 查全部
    } else {
        notices = noticeService.listByCreator(当前用户ID);  // 只查自己的
    }
    return Result.ok(notices);
}
```

**这个写法的问题**：
1. 每个接口都要写 if-else，10 个接口就写 10 遍
2. 哪天角色规则变了（比如新增一个"部门经理"角色能看部门全部数据），你得改几十个文件
3. 新人接手的接口可能忘了写判断 → **安全漏洞**

### 1.3 用数据权限后，你会怎么做？

```java
// Controller：只关心功能权限（能不能用这个功能）
@PreAuthorize("hasAuthority('system:notice:list')")
@GetMapping("/list")
public Result<List<SysNotice>> list() {
    return Result.ok(noticeService.list());  // 不需要写任何 if-else！
}

// Mapper：声明数据权限规则（能看多少数据）
@DataPermission({@DataColumn(key = "userId", value = "create_by")})
List<SysNotice> selectNoticeList();
```

**Controller 不需要写任何权限判断**，SQL 会被自动注入过滤条件。

### 1.4 @PreAuthorize 和 @DataPermission 的关系

很多初学者会困惑：已经有了 `@PreAuthorize`，为什么还要 `@DataPermission`？

| | @PreAuthorize（功能权限） | @DataPermission（数据权限） |
|---|---|---|
| **解决的问题** | 能不能进这个页面？ | 进了之后能看多少数据？ |
| **类比** | 门禁卡——刷卡进楼 | 楼层权限——只能去自己工位那一层 |
| **失败行为** | 返回 403，禁止访问 | 不报错，只返回你有权限看的数据 |
| **作用位置** | Controller 方法 | Mapper 方法 |
| **实现方式** | Spring Security 拦截 | MyBatis 改写 SQL |

**一个具体的例子**：

```
李四访问 /notice/list 接口：

第1步 @PreAuthorize 检查：李四有没有 "system:notice:list" 权限？
  → 有 → 放行，进入 Controller

第2步 @DataPermission 检查：李四的角色能看多少数据？
  → 普通员工角色 → SQL 自动变成：
    SELECT * FROM sys_notice WHERE create_by = '李四的ID'
  → 只返回李四自己创建的 2 条通知
```

**两者是协作关系，不是替代关系**。只用一个是不完整的。

## 第二章：开始写代码

我们的实现分为 6 层，从底层到上层逐步构建：

```
第1层：定义枚举和注解      ← 声明规则
第2层：ThreadLocal 上下文   ← 传递数据
第3层：SQL 条件构建器       ← 拼 WHERE
第4层：MyBatis 拦截器       ← 改写 SQL
第5层：AOP 切面             ← 捕获注解
第6层：注册到 Spring        ← 让系统生效
```

### 2.1 第1层：定义枚举和注解（common-core 模块）

这一层只定义"规则长什么样"，不包含任何实现逻辑。放在 common-core 是为了让所有模块都能引用。

#### 2.1.1 数据范围枚举

```java
// 文件位置: common-core/src/main/java/com/xinghuiTec/enums/DataScopeType.java

package com.xinghuiTec.enums;

import lombok.AllArgsConstructor;
import lombok.Getter;

/**
 * 数据范围类型
 * 决定了一个角色能看到多少数据
 */
@Getter
@AllArgsConstructor
public enum DataScopeType {

    /** 全部数据 —— 超管用，不过滤 */
    ALL("1", "全部数据权限"),

    /** 仅本人 —— 普通员工用，只能看到自己创建的 */
    SELF("5", "仅本人数据权限");

    /** 数据库中 sys_role.data_scope 字段的值 */
    private final String code;

    /** 中文描述 */
    private final String description;

    /**
     * 根据 code 查找枚举
     * 例如数据库存的是 "5"，通过这个方法找到 SELF
     */
    public static DataScopeType findCode(String code) {
        if (code == null || code.isBlank()) {
            return null;
        }
        for (DataScopeType type : values()) {
            if (type.getCode().equals(code)) {
                return type;
            }
        }
        return null;
    }
}
```

**为什么 code 用 "1" 和 "5"？** 这和数据库 `sys_role` 表中已有的 `data_scope` 字段对应。你的数据库表里可能已经有了这个字段，值是 1~6（来源于 RuoYi 框架的设计）。我们只用到 1（全部）和 5（仅本人）。

#### 2.1.2 列映射注解

```java
// 文件位置: common-core/src/main/java/com/xinghuiTec/annotation/datascope/DataColumn.java

package com.xinghuiTec.annotation.datascope;

import java.lang.annotation.*;

/**
 * 定义"占位符 → 数据库列名"的映射
 *
 * 例如：key="userId"  value="create_by"
 * 意思是：SQL 模板中的 #{userId} 最终替换为 create_by 列
 */
@Target(ElementType.METHOD)   // 只能用在方法上
@Retention(RetentionPolicy.RUNTIME)  // 运行时可通过反射读取
@Documented
public @interface DataColumn {

    /** SQL 模板中的占位符名称 */
    String key() default "userId";

    /** 对应的数据库列名 */
    String value() default "create_by";
}
```

**为什么需要这个映射层？** 这是一种解耦设计。如果以后数据库列名从 `create_by` 改成了 `creator_id`，你只需要改注解的 `value`，任何 Java 逻辑都不用动。

#### 2.1.3 数据权限主注解

```java
// 文件位置: common-core/src/main/java/com/xinghuiTec/annotation/datascope/DataPermission.java

package com.xinghuiTec.annotation.datascope;

import java.lang.annotation.*;

/**
 * 数据权限注解 —— 标记在 Mapper 上
 *
 * 用法示例：
 * @DataPermission({@DataColumn(key = "userId", value = "create_by")})
 * List<SysNotice> selectNoticeList();
 *
 * 效果：SQL 自动追加 AND (create_by = '当前用户ID')
 */
@Target({ElementType.METHOD, ElementType.TYPE})  // 方法和类上都能用
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface DataPermission {

    /** 列映射配置数组（支持多列） */
    DataColumn[] value();
}
```

**`value()` 为什么是数组？** 有些复杂场景需要同时过滤多列。例如一个 Mapper 既要按创建人又要按更新人过滤。绝大多数场景只用一个 DataColumn 就够了。

### 2.2 第2层：ThreadLocal 上下文（system-framework 模块）

这一层的目的是**在同一个请求线程内安全地传递 `@DataPermission` 注解**。

**为什么用 ThreadLocal？** 一个 HTTP 请求从 Controller → Service → Mapper → 数据库，全程在同一个线程内执行。ThreadLocal 就像线程的"口袋"，AOP 把注解放进这个口袋，MyBatis 拦截器从口袋里取出来，不需要传参数。

```java
// 文件位置: system-framework/src/main/java/com/xinghuiTec/config/datascope/DataPermissionHelper.java

package com.xinghuiTec.config.datascope;

import com.baomidou.mybatisplus.core.plugins.IgnoreStrategy;
import com.baomidou.mybatisplus.core.plugins.InterceptorIgnoreHelper;
import com.xinghuiTec.annotation.datascope.DataPermission;
import lombok.AccessLevel;
import lombok.NoArgsConstructor;

import java.util.Stack;
import java.util.function.Supplier;

/**
 * 数据权限上下文管理器
 *
 * 核心概念：
 * - PERMISSION_CACHE：存储当前 Mapper 方法的 @DataPermission 注解
 * - REENTRANT_IGNORE：可重入忽略栈，支持嵌套的 ignore() 调用
 */
@NoArgsConstructor(access = AccessLevel.PRIVATE)  // 私有构造，禁止 new
public class DataPermissionHelper {

    // ThreadLocal：每个线程独立的存储空间
    private static final ThreadLocal<DataPermission> PERMISSION_CACHE = new ThreadLocal<>();
    private static final ThreadLocal<Stack<Integer>> REENTRANT_IGNORE = ThreadLocal.withInitial(Stack::new);

    /** AOP 调用：存入注解 */
    public static void setPermission(DataPermission annotation) {
        PERMISSION_CACHE.set(annotation);
    }

    /** MyBatis 拦截器调用：取出注解 */
    public static DataPermission getPermission() {
        return PERMISSION_CACHE.get();
    }

    /** AOP 调用：方法执行完后清理，防止内存泄漏 */
    public static void removePermission() {
        PERMISSION_CACHE.remove();
    }

    // ========== ignore() 机制 ==========
    // 有些场景需要临时关闭数据权限（比如租户管理页面，需要查所有租户）
    // ignore() 方法可以在指定代码块中关闭数据权限

    public static void ignore(Runnable action) {
        enableIgnore();
        try { action.run(); }
        finally { disableIgnore(); }
    }

    public static <T> T ignore(Supplier<T> action) {
        enableIgnore();
        try { return action.get(); }
        finally { disableIgnore(); }
    }

    // ... 内部实现省略，原理是利用 MyBatis-Plus 的 InterceptorIgnoreHelper
}
```

**ignore() 使用示例**：

```java
// 租户管理接口需要查所有租户，不受数据权限限制
@GetMapping("/tenant/list")
public Result<List<SysTenant>> list() {
    List<SysTenant> tenants = DataPermissionHelper.ignore(() ->
        tenantService.list()   // 这段代码里的 SQL 不会被数据权限过滤
    );
    return Result.ok(tenants);
}
```

### 2.3 第3层：SQL 条件构建器（system-framework 模块）

这一层是**核心逻辑**所在——判断当前用户角色，决定要不要加 WHERE 条件、加什么条件。

```java
// 文件位置: system-framework/src/main/java/com/xinghuiTec/config/datascope/PlusDataPermissionHandler.java

package com.xinghuiTec.config.datascope;

import com.xinghuiTec.annotation.datascope.DataColumn;
import com.xinghuiTec.annotation.datascope.DataPermission;
import com.xinghuiTec.constants.TenantConstants;
import com.xinghuiTec.domain.entity.loginUser;
import com.xinghuiTec.enums.DataScopeType;
import com.xinghuiTec.utils.SecurityUtils;
import lombok.extern.slf4j.Slf4j;
import net.sf.jsqlparser.expression.Expression;
import net.sf.jsqlparser.expression.StringValue;
import net.sf.jsqlparser.expression.operators.conditional.AndExpression;
import net.sf.jsqlparser.expression.operators.relational.EqualsTo;
import net.sf.jsqlparser.expression.operators.relational.ParenthesedExpressionList;
import net.sf.jsqlparser.schema.Column;

@Slf4j
public class PlusDataPermissionHandler {

    /**
     * 构建数据过滤 SQL 片段
     *
     * 核心逻辑（按优先级）：
     * 1. 没有 @DataPermission 注解 → 不处理
     * 2. 用户未登录 → 不处理
     * 3. 用户是超管(roleKey=superadmin) → 不处理（看全部）
     * 4. 用户是租户管理员(roleKey=admin) → 不处理（看本租户全部）
     * 5. 普通用户 → 追加 AND (create_by = 'userId')
     */
    public Expression getSqlSegment(Expression where, boolean isSelect) {
        // ===== 第1步：检查有没有注解 =====
        DataPermission annotation = DataPermissionHelper.getPermission();
        if (annotation == null) {
            return where;  // 没有注解，直接返回原 SQL
        }

        // ===== 第2步：获取当前用户 =====
        loginUser loginUser = SecurityUtils.getLoginUser();
        if (loginUser == null) {
            return where;  // 没登录，不处理
        }

        // ===== 第3步：超管/租户管理员 → 不过滤 =====
        if (isSuperAdmin(loginUser)) {
            return where;  // 超管看全部
        }
        if (isTenantAdmin(loginUser)) {
            return where;  // 租户管理员看本租户全部
        }

        // ===== 第4步：普通用户 → 只能看自己创建的 =====
        // 从注解中读取要过滤的数据库列名
        for (DataColumn col : annotation.value()) {
            String columnName = col.value();  // 如 "create_by"
            String userId = loginUser.getUser().getUserId();  // 如 "abc123"

            // 构建等值条件: create_by = 'abc123'
            EqualsTo condition = new EqualsTo(
                new Column(columnName),     // 列名
                new StringValue(userId)     // 值（自动加单引号 + 转义）
            );

            // 用括号包裹: (create_by = 'abc123')
            ParenthesedExpressionList<Expression> parens =
                new ParenthesedExpressionList<>(condition);

            // 和原 SQL 的 WHERE 用 AND 连接
            if (where != null) {
                return new AndExpression(where, parens);
            } else {
                return parens;
            }
        }

        return where;
    }

    // 判断是否超级管理员：检查用户权限列表中是否有 "superadmin"
    private boolean isSuperAdmin(loginUser user) {
        return user.getPermissions() != null
            && user.getPermissions().contains(TenantConstants.SUPER_ADMIN_ROLE_KEY);
    }

    // 判断是否租户管理员：检查用户权限列表中是否有 "admin"
    private boolean isTenantAdmin(loginUser user) {
        return user.getPermissions() != null
            && user.getPermissions().contains(TenantConstants.TENANT_ADMIN_ROLE_KEY);
    }

    public boolean invalid() {
        return DataPermissionHelper.getPermission() == null;
    }
}
```

**为什么不用字符串拼接 SQL，而是用 JSqlParser 的 AST？**

```java
// ❌ 危险：字符串拼接，可能 SQL 注入
String sql = " AND create_by = '" + userId + "'";

// ✅ 安全：JSqlParser 自动转义特殊字符
new EqualsTo(new Column("create_by"), new StringValue(userId));
```

`StringValue` 会自动处理单引号转义，防止 SQL 注入。

### 2.4 第4层：MyBatis 拦截器（system-framework 模块）

这一层负责**在 SQL 执行之前改写它**。

```java
// 文件位置: system-framework/.../config/datascope/PlusDataPermissionInterceptor.java

public class PlusDataPermissionInterceptor
        extends BaseMultiTableInnerInterceptor    // MyBatis-Plus 提供的基类
        implements InnerInterceptor {

    private final PlusDataPermissionHandler handler = new PlusDataPermissionHandler();

    // ===== 拦截查询 =====
    @Override
    public void beforeQuery(Executor executor, MappedStatement ms,
            Object parameter, RowBounds rowBounds,
            ResultHandler resultHandler, BoundSql boundSql) {

        // 检查是否被 ignore() 包裹了
        if (InterceptorIgnoreHelper.willIgnoreDataPermission(ms.getId())) {
            return;
        }
        // 检查有没有 @DataPermission 注解
        if (handler.invalid()) {
            return;
        }
        // 改写 SQL
        PluginUtils.MPBoundSql mpBs = PluginUtils.mpBoundSql(boundSql);
        mpBs.sql(parserSingle(mpBs.sql(), ms.getId()));
    }

    // ===== 拦截更新和删除（防止越权修改别人的数据） =====
    @Override
    public void beforePrepare(StatementHandler sh,
            Connection connection, Integer transactionTimeout) {
        // 同样追加 WHERE 条件
    }

    // ===== 构建 SELECT 的 WHERE =====
    @Override
    protected void processSelect(Select select, int index, String sql, Object obj) {
        Expression segment = handler.getSqlSegment(((PlainSelect)select).getWhere(), true);
        if (segment != null) {
            ((PlainSelect)select).setWhere(segment);
        }
    }

    // ===== 构建 UPDATE 的 WHERE =====
    @Override
    protected void processUpdate(Update update, int index, String sql, Object obj) {
        Expression segment = handler.getSqlSegment(update.getWhere(), false);
        if (segment != null) update.setWhere(segment);
    }

    // ===== 构建 DELETE 的 WHERE =====
    @Override
    protected void processDelete(Delete delete, int index, String sql, Object obj) {
        Expression segment = handler.getSqlSegment(delete.getWhere(), false);
        if (segment != null) delete.setWhere(segment);
    }

    // ===== 抽象方法实现（框架要求） =====
    @Override
    public Expression buildTableExpression(Table table,
            Expression where, String whereSegment) {
        return handler.getSqlSegment(where, true);
    }
}
```

**为什么要拦截 UPDATE 和 DELETE？**

如果不拦截，普通用户发出以下请求会修改别人的数据：

```sql
-- 危险的 SQL（不拦截）
UPDATE sys_notice SET status = '1'

-- 拦截后自动变成（安全）
UPDATE sys_notice SET status = '1' WHERE (create_by = '当前用户ID')
```

### 2.5 第5层：AOP 切面（system-framework 模块）

这一层负责**在 Mapper 方法被调用之前，读取它上面的注解并存到 ThreadLocal**。

#### 2.5.1 切入点（Pointcut）—— 决定"切哪里"

```java
// 文件: system-framework/.../config/datascope/DataPermissionPointcut.java

public class DataPermissionPointcut extends StaticMethodMatcherPointcut {

    @Override
    public boolean matches(Method method, Class<?> targetClass) {
        // 判断1: 方法本身有没有 @DataPermission 注解？
        if (method.isAnnotationPresent(DataPermission.class)) {
            return true;
        }
        // 判断2: 类本身有没有 @DataPermission 注解？
        if (targetClass.isAnnotationPresent(DataPermission.class)) {
            return true;
        }
        // 判断3: Mapper 接口方法上有 @DataPermission 注解？
        // MyBatis 生成的代理类，它的接口上可能带注解
        for (Class<?> iface : targetClass.getInterfaces()) {
            try {
                Method ifaceMethod = iface.getMethod(
                    method.getName(), method.getParameterTypes());
                if (ifaceMethod.isAnnotationPresent(DataPermission.class)) {
                    return true;
                }
            } catch (NoSuchMethodException ignored) {}
        }
        return false;
    }
}
```

**为什么要检查三层？** 注解可以放在不同的位置：

```java
// 情况1: 注解在方法上 → 第1层检查命中
public interface NoticeMapper {
    @DataPermission({@DataColumn(key = "userId", value = "create_by")})
    List<SysNotice> selectList();
}

// 情况2: 注解在类上 → 第2层检查命中（所有方法都继承这个注解）
@DataPermission({@DataColumn(key = "userId", value = "create_by")})
public interface NoticeMapper {
    List<SysNotice> selectList();  // 不用单独写注解，继承类上的
}

// 情况3: MyBatis 动态代理 → 第3层检查命中
// MyBatis 生成的代理对象是 com.sun.proxy.$Proxy123
// 实际注解在 NoticeMapper 接口上
```

#### 2.5.2 通知（Advice）—— 决定"切了做什么"

```java
// 文件: system-framework/.../config/datascope/DataPermissionAdvice.java

public class DataPermissionAdvice implements MethodInterceptor {

    @Override
    public Object invoke(MethodInvocation invocation) throws Throwable {
        // 找到注解
        DataPermission annotation = findAnnotation(invocation);

        // 存入 ThreadLocal（让后面的 MyBatis 拦截器能读）
        DataPermissionHelper.setPermission(annotation);

        try {
            // 执行原始方法（SQL 在这过程中被 MyBatis 拦截器改写）
            return invocation.proceed();
        } finally {
            // 清理 ThreadLocal（防止影响下一个请求）
            DataPermissionHelper.removePermission();
        }
    }
}
```

#### 2.5.3 组装器（Advisor）—— 把 Pointcut 和 Advice 绑在一起

```java
// 文件: system-framework/.../config/datascope/DataPermissionPointcutAdvisor.java

public class DataPermissionPointcutAdvisor extends AbstractPointcutAdvisor {
    private final Advice advice = new DataPermissionAdvice();
    private final Pointcut pointcut = new DataPermissionPointcut();

    @Override public Pointcut getPointcut() { return pointcut; }
    @Override public Advice getAdvice() { return advice; }
}
```

### 2.6 第6层：注册到 Spring（mybatisConfig）

最后一步，把所有组件组装起来。

```java
// 文件: system-framework/.../config/mybatis/mybatisConfig.java

@Configuration
public class mybatisConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor(
            ObjectProvider<TenantLineInnerInterceptor> tenantProvider) {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        // 拦截器 1：多租户（第一位，先过滤租户）
        tenantProvider.ifAvailable(interceptor::addInnerInterceptor);

        // 拦截器 2：数据权限（第二位，在租户数据子集上再过滤）
        interceptor.addInnerInterceptor(new PlusDataPermissionInterceptor());

        // 拦截器 3：分页（第三位，在过滤后的数据集上分页）
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));

        return interceptor;
    }

    // 注册 AOP 切面
    @Bean
    @Role(BeanDefinition.ROLE_INFRASTRUCTURE)
    public DataPermissionPointcutAdvisor dataPermissionPointcutAdvisor() {
        return new DataPermissionPointcutAdvisor();
    }
}
```

**拦截器顺序为什么重要？** 顺序决定了 SQL 被改写的先后：

```
原始 SQL:    SELECT * FROM sys_notice
                ↓
租户插件:    SELECT * FROM sys_notice WHERE tenant_id = '000000'
                ↓
数据权限:    SELECT * FROM sys_notice WHERE tenant_id = '000000' AND (create_by = 'user123')
                ↓
分页插件:    SELECT * FROM sys_notice WHERE tenant_id = '000000' AND (create_by = 'user123') LIMIT 0,10
```

如果数据权限在租户之前，就可能跨租户过滤到不该看到的数据。

## 第三章：如何给角色配置数据范围

### 3.1 数据库配置

数据范围存储在 `sys_role` 表的 `data_scope` 字段：

```sql
-- 查看现有角色
SELECT role_id, role_name, role_key, data_scope FROM sys_role;
```

`data_scope` 的取值含义：

| 值 | 含义 | 适用角色 |
|----|------|---------|
| `"1"` | 全部数据 | 超级管理员 |
| `"5"` | 仅本人数据 | 普通员工 |

### 3.2 创建角色示例

**场景1：创建一个"普通员工"角色，只能看自己创建的数据**

```sql
INSERT INTO sys_role (role_id, role_name, role_key, data_scope, status, create_time)
VALUES (100, '普通员工', 'employee', '5', '1', NOW());
```

给这个角色分配菜单权限后，任何拥有此角色的用户：
- 能看到菜单页面（通过 `@PreAuthorize` 的功能权限）
- 在列表页只能看到自己创建的数据（通过 `@DataPermission` 的数据权限）

**场景2：创建一个"数据管理员"角色，能看全部数据**

```sql
INSERT INTO sys_role (role_id, role_name, role_key, data_scope, status, create_time)
VALUES (101, '数据管理员', 'data_admin', '1', '1', NOW());
```

### 3.3 给用户分配角色

```sql
-- 让用户 user123 拥有"普通员工"角色
INSERT INTO sys_user_role (user_id, role_id) VALUES ('user123', 100);

-- 让用户 user456 拥有"数据管理员"角色
INSERT INTO sys_user_role (user_id, role_id) VALUES ('user456', 101);
```

现在：
- `user123` 查询通知列表 → SQL 自动追加 `AND (create_by = 'user123')` → 只看到自己发的
- `user456` 查询通知列表 → `data_scope='1'` 所以不过滤 → 看到全部

### 3.4 再理解一遍：data_scope 和 roleKey 的关系

在 `PlusDataPermissionHandler` 中，除了检查 `data_scope`，我们还硬编码了两个特殊角色：

```java
// 硬编码特例1：superadmin → 直接不过滤
if (user.getPermissions().contains("superadmin")) {
    return where;
}

// 硬编码特例2：admin（租户管理员）→ 直接不过滤
if (user.getPermissions().contains("admin")) {
    return where;
}

// 其他角色 → 检查 data_scope 字段决定
```

**为什么这样设计？** 超级管理员和租户管理员是系统内置的特殊角色，不管 `data_scope` 字段设什么值，都应该看到全部数据。这是一种"双重保险"——即使有人误把超管的 `data_scope` 改成了 "5"，代码层面也会阻止过滤。

## 第四章：如何在 Mapper 上使用 @DataPermission

### 4.1 基本用法（方法级注解）

```java
public interface SysNoticeMapper extends BaseMapper<SysNotice> {

    /**
     * 查询通知列表
     * @DataPermission 声明：普通用户只能看到自己创建的（create_by = 当前用户ID）
     */
    @DataPermission({@DataColumn(key = "userId", value = "create_by")})
    List<SysNotice> selectNoticeList(SysNoticeQueryDTO query);
}
```

### 4.2 简化写法（类级注解）

如果 Mapper 的所有方法都需要同样的数据权限，可以写在类上：

```java
@DataPermission({@DataColumn(key = "userId", value = "create_by")})
public interface SysNoticeMapper extends BaseMapper<SysNotice> {
    // selectList()  → 自动过滤
    // selectById()  → 自动过滤
    // update()      → 自动过滤
    // delete()      → 自动过滤
}
```

### 4.3 不需要数据权限的 Mapper

不加注解即可，数据权限不会生效，所有用户都能看到全部数据：

```java
// 菜单是全局配置，所有用户都能看到全部菜单
public interface SysMenuMapper extends BaseMapper<SysMenu> {
    // 不写 @DataPermission → 不会被过滤
}
```

### 4.4 不同表用不同的过滤列

```java
// 公告表按 create_by 过滤
@DataPermission({@DataColumn(key = "userId", value = "create_by")})
public interface SysNoticeMapper extends BaseMapper<SysNotice> {}

// 文件表按 upload_by 过滤
@DataPermission({@DataColumn(key = "userId", value = "upload_by")})
public interface SysFileMapper extends BaseMapper<SysFile> {}

// 日志表按 oper_name 过滤
@DataPermission({@DataColumn(key = "userId", value = "oper_name")})
public interface SysOperLogMapper extends BaseMapper<SysOperLog> {}
```

### 4.5 需要注意：不要在不该加的地方加

```java
// ❌ 错误：SysTenantMapper 不应该加数据权限
// 租户管理需要看到所有租户，加了反而看不了
@DataPermission(...)
public interface SysTenantMapper extends BaseMapper<SysTenant> {}

// ✅ 正确：不加注解，配合 DataPermissionHelper.ignore() 使用
public interface SysTenantMapper extends BaseMapper<SysTenant> {}
```

## 第六章：扩展 —— 如何新增一种数据范围

假设以后需要新增"本部门数据"范围，步骤如下：

**第1步：在枚举中新增**

```java
public enum DataScopeType {
    ALL("1", "全部数据权限"),
    SELF("5", "仅本人数据权限"),
    DEPT("3", "本部门数据权限");   // ← 新增
}
```

**第2步：在 Handler 中新增 SQL 构建逻辑**

```java
// PlusDataPermissionHandler 中
if (scopeType == DataScopeType.DEPT) {
    // 查询当前用户所属部门的所有用户ID，构建 IN 条件
    // 例如: create_by IN ('userA', 'userB', 'userC')
    List<String> deptUserIds = getUserIdsByDept(loginUser);
    // ... 构建 IN 表达式
}
```

**第3步：在数据库 sys_role 中配置新角色**

```sql
INSERT INTO sys_role (role_id, role_name, role_key, data_scope, ...)
VALUES (200, '部门经理', 'dept_manager', '3', ...);
```

**第4步：在需要部门过滤的 Mapper 上加注解**

```java
@DataPermission({@DataColumn(key = "userId", value = "create_by")})
List<SysNotice> selectNoticeList();
```

注解不用改！因为 `@DataColumn` 只是声明"用什么列过滤"，具体 SQL 逻辑由 Handler 根据 `DataScopeType` 决定。

## 第七章：常见问题

### Q1: 为什么不把 data_scope 判断放在注解里？

如果注解里可以写 `scope = "SELF"`，那每个 Mapper 都要手动指定用哪种范围。但数据范围应该由**角色配置**决定（超管看全部、员工看自己），而不是由**代码**决定。代码只需要声明"这个 Mapper 需要数据过滤"，具体怎么过滤由登录用户的角色决定。

### Q2: 超管为什么不过滤？在哪里判断的？

在 `PlusDataPermissionHandler.getSqlSegment()` 中：
```java
if (isSuperAdmin(loginUser)) {
    return where;  // 原样返回，不追加任何条件
}
```

### Q3: 如果某个接口临时需要查全部数据怎么办？

使用 `DataPermissionHelper.ignore()` 包裹：

```java
List<SysNotice> allNotices = DataPermissionHelper.ignore(() ->
    noticeMapper.selectList(null)
);
```

### Q4: 多表 Join 查询能用吗？

当前版本不支持多表 Join。如果确实需要，需要用 `DataPermissionHelper.ignore()` 包裹，然后手动写过滤条件。

### Q5: 数据权限和租户隔离有什么区别？

```
租户隔离：不同公司的人完全隔离，看不到对方的数据
数据权限：同一家公司内，不同角色的人看到不同范围的数据

SQL 顺序：
1. 租户插件: WHERE tenant_id = '公司A'
2. 数据权限: AND (create_by = '员工张三')
```
